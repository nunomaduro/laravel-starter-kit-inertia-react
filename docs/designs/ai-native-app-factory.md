## Vision

### 10x Check
"An AI co-founder for freelance developers." Not just scaffolding — an AI that understands the business domain, generates the app, then *stays with the project* as an ongoing development partner. It reads the codebase, suggests features, writes migrations, generates tests, and handles client change requests. The freelancer goes from "solo dev building everything" to "CEO managing an AI engineering team." This transforms the value from "save 3 months of setup" to "10x your capacity permanently."

### Platonic Ideal
1. **Magic on first install.** `composer create-project cogneiss/factory my-hr-app` → beautiful, functional HR system running in 30 seconds.
2. **AI knows everything.** Queries data ("how many employees took PTO?") AND understands code ("add a performance review module" → writes the migration, model, pages, routes).
3. **Modules compose like LEGO.** Install HR + Projects → AI automatically understands cross-module relationships without configuration.
4. **Updates are invisible.** Module updates run in background. Breaking changes flagged. Migrations auto-generated. *(Deferred — not in any current phase. Revisit post-Phase 3.)*
5. **Developer never leaves the terminal.** Every interaction works from artisan commands AND web UI with identical capability.

## Strategic Decisions

### Business Model
Kit is DIY, Platform is managed. Phase 1 kit ($299) is explicitly the "build it yourself" tier. Phase 3 platform ($49/mo) is the "we handle the complexity" tier. Different buyers, both valid.

### Implementation Strategy
Sequential Phases (A→B→C). Build Phase 0→1→2→3 in order. Each phase gates the next. Validates demand before building the differentiator.

### Phase 2 Additions
- **Spike (first 2 days):** Prototype `ModuleRelationshipRegistry` with HR + Projects test case. Validate that cross-module queries and AI context work before committing to full build. If spike fails, fall back to independent modules with manual relationship configuration.
- Cross-module intelligence: `ModuleRelationshipRegistry` — each module can declare typed relationships to other modules:
  ```php
  interface DeclaresModuleRelationships {
      /** @return array<ModuleRelationship> */
      public function relationships(): array;
  }

  final readonly class ModuleRelationship {
      public function __construct(
          public string $sourceModel,   // e.g., 'projects::task'
          public string $targetModel,   // e.g., 'hr::employee'
          public string $type,          // 'belongsTo', 'hasMany', 'morphMany'
          public string $foreignKey,
      ) {}
  }
  ```
  The `ModuleProvider` base class auto-registers relationships at boot. AI Context Provider reads the registry to include cross-module relationships in its system prompt. **Graceful degradation:** If target module isn't installed, skip the relationship and log a warning. Zero crashes.

### Phase 3a Additions
- AI Codebase Assistant (MVP): AI answers questions about the project's models, routes, and actions using existing `laravel/mcp` tool definitions. Does NOT do full codebase embedding or AST parsing — it uses the MCP server's route listing, model introspection, and action discovery tools that already exist. Code generation scoped to: adding fields to models, creating simple actions, generating migration stubs.
- **Security:** AI-generated code is NEVER auto-applied. All code suggestions are presented as diffs for developer review. The AI runs in a read-only context — it can query models and routes but cannot execute migrations or modify files without explicit developer approval via the CLI.
- **Prompt injection defense-in-depth:** (1) Input sanitization strips instruction-override patterns, (2) MCP tools enforce tenant scope regardless of prompt, (3) Output filter catches cross-tenant data leaks.

## Security Considerations

### AI Codebase Assistant (Phase 3a)
- **Sandboxing:** AI operates in read-only mode. Can query models, routes, actions via MCP tools. Cannot execute code, run migrations, or modify files.
- **Review gate:** All AI-generated code presented as diffs. Developer must explicitly approve via CLI (`--apply` flag) or web UI (approve button).
- **Permissions model:** AI assistant respects tenant boundaries. Can only access models/routes within the current organization context.
- **Rate limiting:** AI queries rate-limited per user (100/hour) and per organization (500/hour) to prevent abuse.
- **Audit trail:** All AI code suggestions logged with timestamp, user, organization, prompt, and response. Stored via activity logging.
- **Prompt injection:** Defense-in-depth with input sanitization, tool-level tenant enforcement, and output filtering.

### Module System (Phase 2)
- Modules installed via Composer (signed packages when available)
- Module installer runs pre-flight security check: no eval(), no shell_exec(), no file_put_contents outside expected paths
- Each module's migrations reviewed by installer for destructive operations (DROP TABLE, TRUNCATE)
