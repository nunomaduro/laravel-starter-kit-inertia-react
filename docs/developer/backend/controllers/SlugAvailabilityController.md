# SlugAvailabilityController

## Purpose

JSON endpoint that checks whether a given organization slug is available, reserved, or already taken. Used by the workspace URL settings page for debounced real-time availability feedback.

## Location

`app/Http/Controllers/Api/SlugAvailabilityController.php`

## Method

| Method | HTTP | Route | Auth | Purpose |
|--------|------|-------|------|---------|
| `__invoke` | GET | `/api/slug-availability` | auth required | Check slug availability |

## Route

- `api.slug-availability`: `GET /api/slug-availability?slug={slug}`

## Request

| Query Parameter | Description |
|----------------|-------------|
| `slug` | The slug string to check |

The current org (from `TenantContext`) is automatically excluded from the taken check, so the current org's own slug always appears available.

## Response

```json
{
  "available": true,
  "reserved": false,
  "taken": false,
  "suggestion": null
}
```

| Field | Type | Description |
|-------|------|-------------|
| `available` | bool | `true` if the slug can be used |
| `reserved` | bool | `true` if slug is in the `reserved-slugs` config list |
| `taken` | bool | `true` if another organization already uses this slug |
| `suggestion` | string\|null | Alternative slug (e.g. `my-org-1`) when taken; `null` if available |

## Slug Validation

Slugs must match `/^[a-z0-9][a-z0-9\-]{1,61}[a-z0-9]$/`. Invalid format returns `available: false` immediately without a DB query.

When taken, suggestions are generated by appending `-1` through `-10` and returning the first unclaimed candidate.

## Related Components

- **Frontend**: `resources/js/pages/settings/general.tsx` — calls this via `fetch()` with 500ms debounce
- **Rule**: `app/Rules/SlugAvailable.php` — server-side validation using the same logic
- **Config**: `config/reserved-slugs.php`
